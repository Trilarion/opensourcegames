# Code Architecture

This folder contains all Python scripts and utilities to maintain the Open Source Games List database.

## Architecture Overview

### Core Modules (`utils/`)

- **`osg.py`**: Main game/entry operations (reading/writing/parsing entries, developers, inspirations)
- **`osg_parse.py`**: Lark grammar-based parsing for markdown entry files and listing files
- **`constants.py`**: Centralized paths, configuration, validation constants, and lookup tables
- **`utils.py`**: Generic utilities (file I/O, subprocess, archive handling)
- **`archive.py`**: Git repository archiving helpers
- **`osg_github.py`**: GitHub API integration (PyGithub)
- **`osg_gitlab.py`**: GitLab API integration (Python-GitLab)
- **`osg_wikipedia.py`**: Wikipedia search integration
- **`osg_statistics.py`**: Statistics calculation and visualization
- **`osg_rejected.py`**: Rejected entries management
- **`osg_ui.py`**: Qt/PySide6 UI utilities

### Maintenance Scripts

- **`maintenance_entries.py`**: Validate and maintain entry files (TODO: many consistency checks)
- **`maintenance_developers.py`**: Maintain `developers.md`, sync with entries
- **`maintenance_inspirations.py`**: Maintain `inspirations.md`, sync with entries
- **`maintenance_screenshots.py`**: Update screenshot metadata and verify integrity

### Data Import/Export

- **`archive_update.py`**: Clone/update git repositories in local archive
- **`git_statistics.py`**: Extract statistics from archived repositories
- **`synchronization/`**: Import scripts for external sources (GitHub, GitLab, LibreGameWiki, SourceForge, Codeberg)

### Grammars

- **`grammar_entries.lark`**: Lark grammar for parsing entry markdown files
- **`grammar_listing.lark`**: Lark grammar for parsing developer and inspiration listing files

## Setup & Dependencies

### Requirements
- Does not need to be compatible with older Python versions. Currently runs with Python 3.12+
- Dependencies listed in `requirements.txt` (unpinned for flexibility)

### Install Dependencies
```bash
pip install -r requirements.txt
```

### Configuration
- Optional: Create `local-config.ini` for local overrides
- GitHub/GitLab tokens can be stored in `private.properties` (not tracked in Git)

## Running Maintenance Scripts

Each maintenance script provides an interactive UI with available operations:

```bash
# Maintain game entries
python maintenance_entries.py

# Maintain developer information
python maintenance_developers.py

# Maintain inspiration information
python maintenance_inspirations.py

# Update/sync screenshot metadata
python maintenance_screenshots.py

# Clone/update archived repositories
python archive_update.py

# Generate statistics
python git_statistics.py
```

## Data Flow

1. **Entry files** (`../entries/*.md`) → parsed by `osg_parse.py` → internal dictionary representation
2. **Listing files** (`../developers.md`, `../inspirations.md`) → parsed by `osg_parse.py`
3. **External APIs** (GitHub, GitLab, Wikipedia) → enriched via `osg_github.py`, `osg_gitlab.py`, etc.
4. **Git repositories** → archived locally via `archive_update.py` → statistics generated by `git_statistics.py`
5. **Generated files** → `../docs/data.json`, `../statistics.md`, `../entries/screenshots/README.md`

## Code Style & Conventions

### Docstring Style
- Follow **PEP 257** conventions, every function should have a docstring
- Multi-line docstrings for all functions/classes with detailed explanation with triple """ and comments on the next line
- Include parameter descriptions for non-obvious parameters
- Avoid comments with multiple = or - as lines

### Naming Conventions
- **Functions/variables**: `snake_case`
- **Classes**: `PascalCase` (e.g., `DevelopersMaintainer`, `EntryTransformer`)
- **Constants**: `UPPER_SNAKE_CASE` (e.g., `HEIGHT = 128`)
- **Private members**: Prefix with `_` (e.g., `_internal_state`)
- **Variable names**: Use expressive, extended variable names. (e.g. `number`, `language`)

### Code Organization
- Keep modules focused and single-purpose
- Utility functions in `utils/` (general-purpose, no OSG-specific logic)
- OSG-specific operations in `utils/osg*.py`
- Parsing logic isolated in `osg_parse.py` using Lark grammars
- Constants centralized in `constants.py`, it also acts as the central configuration file (except for credentials and local paths)

### Error Handling & Validation
- Raise `RuntimeError` for logical errors or validation failures
- Include descriptive error messages referencing the problematic entry/developer
- Use assertions for internal logic that should never fail (e.g., `assert len(x) <= 1`)
- Print informative messages for maintenance operations (users run these interactively)

### Data Structures
- Parse markdown into dictionaries with field names as keys
- Convert between list (for iteration/sorting) and dictionary (for lookup) formats as needed
- Use `str.casefold()` for case-insensitive string comparisons
- Leverage `difflib.SequenceMatcher` for fuzzy name matching (similarity ratio)

### File I/O
- All text files are UTF-8 encoded with `errors='ignore'` for robustness
- Use `pathlib.Path` for all path operations (all paths in `constants.py`)
- Read/write operations centralized in `utils.py` and `osg.py`

### Dependencies
- `requirements.txt` is intentionally unpinned to allow flexible version updates
- Test compatibility with Python 3.12+ before committing changes

### Comments
- Add comments for complex logic, workarounds, or non-intuitive patterns
- Mark TODO items for future improvements (searchable in maintenance scripts)

### Testing
- No formal test suite currently exists
- Manual testing via maintenance script UIs is the primary validation
- Verify consistency checks and data integrity after changes

## Common Tasks

### Adding a New Consistency Check
1. Add TODO in `maintenance_entries.py`
2. Implement as a method in `EntryMaintainer` class (or create new class)
3. Add action to the interactive UI dictionary
4. Test with sample entry data

### Modifying Entry Format
1. Update grammar in `grammar_entries.lark`
2. Update `constants.py` field definitions
3. Update parsing in `osg_parse.py` if needed
4. Run `maintenance_entries.py` to validate all entries

### Adding External Data Source
1. Create new import script in `synchronization/`
2. Use existing patterns (GitHub, GitLab, LibreGameWiki)
3. Ensure data is normalized to internal entry format
4. Document in this README

## Troubleshooting

- **Import errors**: Ensure `requirements.txt` is installed: `pip install -r requirements.txt`
- **API rate limits**: GitHub/GitLab scripts may hit rate limits; credentials can help
- **Encoding issues**: Most scripts handle UTF-8
